## Бизнес-правила взвешивания
- Вагоны взвешиваются потележечно: 1-я тележка, затем 2-я.
- После второй тележки программа суммирует массу вагона и сохраняет результат.
- `TrainTime` присваивается при первой тележке первого вагона состава и одинаков для всех вагонов состава.
- `WagonTime` присваивается при второй тележке каждого вагона.
- Нумерация вагонов идет внутри состава и сбрасывается кнопкой завершения состава.
- В таблицах взвешивания лимит отображаемых вагонов убран: показываются все взвешенные вагоны текущего режима.
- Превышение НПВ подсвечивается и блокирует перенос в Firebird.
- Дискретность применяется к отображаемой и сохраняемой массе.
- На динамической форме после сохранения вагона показатели тележек сбрасываются, чтобы не оставались данные предыдущего вагона.
- Динамическое взвешивание использует активную строку калибровки; нулевой коэффициент `0` считается валидным значением.

## Настройки и сервис
- Локальные настройки хранятся в `settings.json` рядом с исполняемым файлом.
- Если `settings.json` отсутствует, он создается автоматически.
- Значения по умолчанию:
  - рабочий COM-порт АЦП: `COM1`;
  - НПВ: `140 т`;
  - дискретность: `0.05 т`;
  - лимит нуля весовщика: `2% НПВ`;
  - административный лимит нуля: `100% НПВ`.
- Продовый COM-порт: `COM1`.
- Дев-схема с виртуальными портами: основная программа обычно на `COM3`, эмулятор на `COM4`; в системе должна быть программа проброса виртуальных COM-портов.
- Пароль администратора хранится только как PBKDF2 hash + salt, без plaintext.
- `MainForm` берет рабочий COM-порт из `settings.json`.
- `ServiceForm` разделена на сервисные режимы статики и динамики.
- На каждой вкладке сервиса, где нужен АЦП, есть локальное управление подключением COM-порта.
- При переходе между вкладками сервисной формы конфликтующий reader закрывается автоматически:
  - вкладки статики закрывают динамический reader;
  - вкладки динамики закрывают статический reader;
  - вкладки без АЦП закрывают оба reader-а.
- При закрытии сервисной формы открытые COM-порты статики и динамики закрываются.
- Переключение активного канала CH0/CH1 доступно весовщику как аварийное переключение и логируется.

## АЦП и протокол
- АЦП возвращает оба канала: CH0 и CH1.
- Выбор канала означает выбор используемого значения, а не выбор запрашиваемого канала.
- COM-порт: 4800 бод, Even parity, 8 бит, 1 стоп-бит.
- Статический режим:
  - запрос poll byte `214`;
  - ответ 4 байта;
  - `CH0 = byte[1] * 256 + byte[0]`;
  - `CH1 = byte[3] * 256 + byte[2]`.
- Динамический режим:
  - установка потока `Set=126`, `Req=254`;
  - непрерывный поток 5-байтовых сэмплов;
  - рабочая схема сэмпла: `CH0_LO CH0_HI CH1_LO CH1_HI AUX`;
  - `CH0 = byte[1] * 256 + byte[0]`;
  - `CH1 = byte[3] * 256 + byte[2]`;
  - `AUX` хранится как диагностический байт.
- По тестам усреднений динамики стабильными оказались `Avg=64` и `Avg=128`; проверка других усреднений была отдельным этапом.
- Таймаут живой связи:
  - `StaticWeighingForm`: `2000` мс;
  - `DynamicWeighingForm`: `5000` мс;
  - `ServiceForm` static reader: `1000` мс;
  - `ServiceForm` dynamic reader: `5000` мс.
- Динамический reader пытается восстановить поток через `Req`/`Set` с периодом около 5 секунд.
- При открытии COM-порта очищаются входной и выходной буферы.

## Динамика и диагностика зависаний
- Для динамического сервиса добавлена диагностика в существующий аудит `EventId=2012`.
- Диагностика сервисной динамики пишет `ObjectType=AdcDynamicService` и поля:
  - `bytes`, `samples`, `skipped`;
  - `samplePosted`, `sampleApplied`, `samplePending`;
  - `rawPosted`, `rawApplied`, `rawPending`;
  - `logAppended`, `logMaxMs`.
- Цель диагностики - доказать причину зависания UI, а не менять поведение по догадке.
- При прошлых зависаниях сервисной вкладки поток продолжал идти, но UI переставал отвечать; по логам частота была примерно `160` сэмплов за 5 секунд, то есть около `32 Гц`.
- Оценка проблемного объема: примерно 30 минут работы, около `57 600` сэмплов, около `115 000` UI-событий из-за `SampleReceived` + `RawSampleReceived`.
- Связь между `Сервисный режим Динамика` и `Калибровка Динамика` полностью разведена (2026-07-01, см. `docs/dynamic-service-coupling-checklist.md` и `docs/dynamic-service-current-dataflow.md`): у каждой вкладки свой reader (`_dynamicServiceSim` / `_dynamicCalibSim`), свои подписки на события, свой connection workflow и свои диагностические счётчики; raw-подписки у калибровки нет.

## Проблема GC-давления сервисного лога (решено 2026-07-03)
- Симптом: на вкладке `Сервисный режим Динамика` при включённой галке лога GC работал непрерывно, аллокации росли неограниченно во времени (в поздней фазе ~`520 МБ/с`, `gen0` ~`24/с`). При выключенной галке лога аллокации прекращались — это указало, что источник в log-on пути, а не в reader-е.
- Диагностика (использовалась и зафиксирована в коммите `9bfec02`, затем снята в `4116efd`): в аудит `EventId=2012` временно добавляли аллокационные метрики через `GC.GetAllocatedBytesForCurrentThread` (брекеты по сегментам), `GC.GetTotalAllocatedBytes`, `GC.CollectionCount(0)` — поля `totalAllocKB/gen0/allocReaderKB/allocLogKB`. Именно так локализовали.
- Root cause: живой лог был на `RichTextBox`. Обрезка контента (сначала через `.Lines`/`GetFirstCharIndexFromLine`, потом через собственный посимвольный учёт) **не удаляла текст** из-за ненадёжной индексации RichTextBox (нормализация `\r\n`/`\n`, накопление undo-буфера). Контент рос неограниченно, и каждая вставка сверху переформатировала весь текст.
- Решение (коммит `9bfec02`): лог переведён на **owner-drawn `ListBox` (`_lstDynamicLog`)** — рисуются только видимые строки, контент ограничен числом элементов (`DynamicServiceLogLineLimit = 300`), нет RTF/undo/переформатирования. Дополнительно: живые CH0/CH1 throttl-ятся таймером (последний сэмпл раз в 100 мс), строка лога форматируется через переиспользуемый `StringBuilder` без LINQ.
- Подтверждено на длинном прогоне: `totalAlloc` ~`2.6 ГБ/5с` → ~`1 МБ/5с` (~2500×), `gen0` ~`121/5с` → ~`0`, плоско без роста; `logMaxMs` до `537` мс → `0`.
- Важный вывод на будущее: `RichTextBox` не годится как живой лог высокой частоты (~30 Гц) — использовать owner-drawn `ListBox`/виртуализированный контрол.

## Калибровка
- Статическая калибровка хранится по точкам и поддерживает историю снятия точек через `deleted_at`.
- Снятые строки статической калибровки подсвечиваются и становятся read-only.
- В гриде статической калибровки колонка `Активна` перенесена в первую позицию и показывается текстом `Да`/`Нет` (read-only), как на динамике; чекбокс убран (он и раньше был нередактируемым). Множественные активные точки сохранены — они нужны для расчёта прямой по МНК, поэтому кнопки `Добавить строку`/`Удалить выбранную` остались.
- Грид статической калибровки дополнен колонками `Добавлена`/`Снята` (`CalibPoint.CreatedAt`), см. раздел «Обновление на 2026-07-09».
- `CalibrationCalculator.Convert` возвращает `null` при отсутствии активных точек калибровки канала (отличает "не откалибровано" от легитимного нуля); `StaticWeighingForm` при этом сохраняет прежний откат к `0`.
- Динамическая калибровка хранится в `calibration_dynamic` историческими строками:
  - `is_active` определяет активную строку;
  - `deleted_at` фиксирует снятие старой строки;
  - `KPlus = 0` и `KMinus = 0` являются валидными коэффициентами.
- Для динамической калибровки добавлена миграция `migrations/004_calibration_dynamic_history.sql`.
- Для истории точек статической калибровки добавлена миграция `migrations/005_calibration_points_created_at.sql`.
- `LocalRepository.SaveDynamicCalibAsync` создает новую активную строку и снимает старую; `updated_at` для `calibration_dynamic` не используется.
- На вкладке `Калибровка Динамика` можно сохранять отдельно `K→` или `K←`; пустое поле сохраняет текущее активное значение второго коэффициента.
- Снятые строки динамической калибровки подсвечиваются как на статике.
- На вкладке динамической калибровки показываются текущий код АЦП, текущий вес и статус подключения динамического АЦП.

## Корректировки
- `CorrectionsForm` работает с локальной PostgreSQL и Firebird системы учета.
- Контекст `Состав` / `Сутки` реализован.
- Форма при открытии остается пустой; загрузка выполняется только кнопкой `Обновить`.
- `Обновить` всегда применяет активный контекст к обеим таблицам: локальной PostgreSQL и Firebird.
- В режиме `Состав` используется дата + время состава; в режиме `Сутки` используется только дата, поле времени не участвует в запросе.
- При сохранении/переносе вагона режим тара/брутто больше не сбрасывается на брутто после каждой операции; при открытии формы выставляется брутто, дальше оператор управляет режимом сам.
- Правый grid корректировки сортируется без некорректного `ORDER BY`, чтобы не ловить Firebird `invalid ORDER BY clause`.
- Grid корректировки растягивается по ширине; минимальная ширина формы сохранена, чтобы раскладка не ломалась.

## Печать отвесной
- `PrintForm` выбирает данные из Firebird по фильтрам и формирует PDF через QuestPDF.
- Верхние две панели формы печати переведены на стабильную растягиваемую раскладку через `TableLayoutPanel` и корректные `Anchor/Dock`.
- Нижняя панель формы печати также переведена на `TableLayoutPanel`; поля и кнопка тянутся по ширине.
- Нижние лейблы формы печати используют `UiColors.TextMuted`, потому что статусная панель светлая.
- В grid формы печати убраны лишние `тара док` и `ндок`; колонка `№п/п` поставлена последней.
- В печатном PDF колонка `№п/п` выровнена влево, чтобы порядковый номер был ближе ко времени и дальше от номера вагона.

- `AuditLogger` работает через ограниченную очередь на `30000` событий и фоновый писатель.
- При недоступной БД программа продолжает показывать вес; журнал аудита копится в памяти до лимита, старые события при переполнении теряются.

## Обновление на 2026-07-09

### Калибровка Статика
- `ServiceForm`: вкладка `Калибровка Статика` получила отдельный экземпляр `SimA04ReaderStatic`, независимый от вкладки `Сервисный режим Статика` (по аналогии с ранее сделанной развязкой динамики) — своё UI подключения, свои подписки, взаимное исключение reader-ов при переключении вкладок.

## Проверки и ограничения окружения
- Пользователь явно запретил запускать `build`, `restore`, `clean` для любых правок.
- На текущем Linux-хосте полноценная сборка WinForms-проекта не является рабочей проверкой из-за WindowsDesktop/WinForms-зависимостей.
- Перед коммитами проверять `git status --short`, так как рабочее дерево может содержать пользовательские изменения.
- Не откатывать чужие незакоммиченные изменения без прямого запроса.
- Форматирующие сдвиги пользователя не трогать без отдельного запроса.

## Что важно помнить дальше
- `settings.json` является локальной конфигурацией рабочего места и не должен коммититься как пользовательский файл.
- Не возвращать `.resx`-шум `MainForm.resx` и `ScaleListener/MainForm.resx`; ранее они были исключены через `.gitignore`.
- Для WinForms Designer возможны изменения порядка строк и служебного кода; перед коммитом смотреть содержательный diff.
- Предпочтительно делать точечные UI-правки в рамках текущих форм, без широких рефакторингов.
- В динамической калибровке признак заданности - активная строка, а не ненулевое значение коэффициента.
- Нулевой коэффициент динамики является осознанным валидным значением.
