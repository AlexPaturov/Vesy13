# CorrectionsForm: контекст работы по составу / суткам

## Цель
- Сделать `CorrectionsForm` контекстной: пользователь работает либо с одним составом, либо со всем днём.
- Обновление всегда должно синхронно применять активный контекст к локальной БД и Firebird.
- При открытии формы данные не должны подгружаться автоматически.

## UI-режимы
- Добавить два взаимоисключающих радиобаттона:
  - `rbModeTrain` - режим "Состав"
  - `rbModeDay` - режим "Сутки"
- Режим по умолчанию:
  - `Состав`, если нет отдельного требования от пользователя
- Поведение при переключении режима:
  - `rbModeTrain` выбран - пикер времени активен
  - `rbModeDay` выбран - пикер времени неактивен и визуально подчеркнут как disabled
  - пикер даты остаётся активным всегда
- Все необходимые контролы уже добавлены на форму:
  - `rbTrainMode`
  - `rbDayMode`
  - `_dtpTrainDate`
  - `_dtpTrainTime`
- Запрещено:
  - создавать новые компоненты без отдельного согласования
  - менять позиционирование элементов на форме без предварительного согласования

## Поведение пикеров
- `DateTimePicker` даты:
  - формат только дата
  - `CustomFormat = "dd.MM.yyyy"`
  - `Format = Custom`
- `DateTimePicker` времени:
  - формат только время
  - `CustomFormat = "HH:mm:ss"`
  - `Format = Custom`
  - `ShowUpDown = true`
- Для режима `Сутки` значение времени либо игнорируется, либо не влияет на запросы.

## Логика кнопки `Обновить`
- Кнопка `Обновить` остаётся единственной точкой запуска загрузки.
- `Обновить` всегда читает текущий активный контекст:
  - `Состав` -> дата + время
  - `Сутки` -> только дата
- При смене режима или даты/времени пользователь нажимает ту же кнопку `Обновить`.
- Никакой второй кнопки для выбора/применения контекста не добавлять.

## Что должно загружаться
- Режим `Состав`:
  - локальная база: только вагонный набор для выбранного `TrainTime`
  - Firebird: только записи для выбранных `DT + VR`
- Режим `Сутки`:
  - локальная база: все составы за выбранную дату
  - Firebird: все записи за выбранную дату

## Изменения в форме
- Убрать автозагрузку данных из `OnLoad`.
- Оставить форму пустой при открытии.
- Хранить выбранный контекст в полях формы:
  - выбранная дата
  - выбранное время
  - текущий режим (`Train` / `Day`)
- При смене режима не менять данные молча:
  - пользователь меняет режим
  - затем жмёт `Обновить`
- Не добавлять новых визуальных элементов, кроме уже существующих контролов, если это не согласовано отдельно.
- Не перестраивать layout формы без предварительного согласования изменений.

## Изменения в репозиториях
- `LocalRepository`:
  - добавить выборку по одному составу
  - добавить выборку всех составов за день
- `FactoryRepository`:
  - добавить выборку по одному составу
  - добавить выборку всех записей за день
- Сигнатуры запросов должны принимать только нужный контекст:
  - для `Состава` - дата + время
  - для `Суток` - только дата

## Синхронизация UI и данных
- Активный режим должен быть виден пользователю всегда.
- Контекст отображения и контекст запроса должны совпадать.
- Если выбран режим `Сутки`, поле времени должно быть disabled и визуально показывать неактивное состояние.
- Если выбран режим `Состав`, поле времени должно быть активным и участвовать в запросе.

## Ограничения реализации
- Не добавлять отдельную кнопку "Открыть состав".
- Не разносить логику по двум разным сценариям обновления.
- Не менять существующую идею: одна кнопка `Обновить`, один активный контекст.
- Стараться не ломать текущую размерку формы:
  - использовать уже существующий верхний блок
  - добавлять минимально необходимое количество контролов
- Не создавать новые элементы UI без отдельного разрешения.
- Не менять позиционирование элементов на форме без предварительного согласования.
