# Аппаратная интеграция

## АЦП "СИМ А04"

Статический драйвер находится в `Services/Hardware/SimA04ReaderStatic.cs`.
Динамический драйвер находится в `Services/Hardware/SimA04ReaderDynamic.cs`.

Статический драйвер работает по циклу:

1. открыть COM-порт;
2. отправить один poll-байт;
3. принять ответ;
4. подождать 20 мс после последнего принятого байта;
5. разобрать накопленный буфер как кадр;
6. опубликовать события;
7. отправить следующий poll-байт.

## Обработка входного потока

Фильтры — отдельная стадия между драйвером и формой взвешивания, не часть драйвера:
`Services/Hardware/Filters/StaticFilterPipeline.cs` и `DynamicFilterPipeline.cs`. Пайплайн
подписывается на события драйвера, прогоняет коды через фильтры и публикует собственные
события `FilteredFrameReceived` / `FilteredSampleReceived`; форма подписана на них
(`StaticWeighingForm.cs`, `DynamicWeighingForm.cs`). Сырые события и `ConnectionChanged`
формы берут напрямую у драйвера, минуя пайплайн.

Логика статики и динамики намеренно продублирована — общего кода фильтров между ними нет.

### Разбор кадра

Статика (`SimA04Frame.Parse`, `Models/SimA04Frame.cs`): проверяется только длина пакета
(ровно 4 байта). Контрольной суммы в статическом протоколе нет.

Динамика (`SimA04ReaderDynamic.cs`, `SimA04DynamicSample.Parse`): скользящее окно 5 байт с
контрольной суммой `AUX = (CH0_LO + CH1_LO + AuxOffset) & 0xFF`. Сэмпл с несошедшейся суммой
не публикуется, окно сдвигается на один байт (`ShiftLeft`), счётчик `SkippedBytes`
увеличивается — так поток ресинхронизируется после сбоя. Драйвер поднимает
`ChecksumRejected`, пайплайн пишет по нему аудит `2023`.

### Цепочки фильтров

| | Порядок |
| --- | --- |
| Статика | клэмп → дельта → EMA |
| Динамика | клэмп → застревание → дельта → EMA |

Порядок в динамике не косметический: застревание проверяется **до** дельта-фильтра, потому
что у застрявшего кода дельта нулевая и дельта-фильтр принял бы его за идеально стабильный
сигнал.

- **Клэмп** — код вне `[min..max]` бракуется. Границы задают физически возможный диапазон:
  ловит обрыв датчика и мусор в линии.
- **Дельта** — бракуется код, отличающийся от предыдущего **принятого** больше чем на порог.
  Предыдущее значение обновляется только принятым кодом, иначе одиночный выброс стал бы
  точкой отсчёта для следующей дельты и утащил бы фильтр за собой.
- **Застревание** (только динамика) — считает строго равные коды подряд, бракует начиная с
  порогового. Живой тензодатчик всегда шумит младшими разрядами, поэтому длинная серия
  бит-в-бит одинаковых кодов означает мёртвый канал. Счётчик сбрасывается сам, как только
  код изменился.
- **EMA** — единственный фильтр, который не бракует, а меняет значение:
  `ema = α × код + (1 − α) × предыдущая_ema`. Чем меньше `α`, тем сильнее сглаживание и
  больше запаздывание.

### Отбраковка

Отбракованный кадр не публикуется вообще — форма его не видит. Отдельного «плохого» состояния
для этого не заводится: если сбой длится дольше `ConnectionTimeoutMs`, watchdog драйвера сам
переводит связь в «нет валидного потока», и формы блокируют взвешивание уже существующей
защитой.

Решение об отбраковке принимается **по активному каналу** — в вес идёт только он
(`ActiveCode`). Срабатывание на неактивном канале пишется в аудит, но кадр не бракует.

Состояние фильтров (предыдущий код, накопленная EMA, счётчик застревания) — своё на каждый
канал.

### Настройки и аудит

Все фильтры по умолчанию выключены, границы клэмпа инертны (`0..65535`). Реальные границы из
кода не выводятся — их подбирают на живом железе (ненагруженные весы дают ≈3589).

Параметры живут в `settings.json` (`AppSettings`), правятся на вкладке `Настройки` формы
`ServiceForm`, валидируются в `SettingsService.EnsureDefaults`. Пайплайн читает поля на каждом
кадре, поэтому сохранение действует без перезапуска.

| Фильтр | Включение | Параметры | Аудит |
| --- | --- | --- | --- |
| Клэмп | `StaticClampEnabled` / `DynamicClampEnabled` | `*ClampMinCode`, `*ClampMaxCode` | `2020` |
| Дельта | `StaticDeltaEnabled` / `DynamicDeltaEnabled` | `*DeltaMaxCodes` | `2021` |
| Застревание | `DynamicStuckEnabled` | `DynamicStuckSamples` | `2022` |
| Контрольная сумма | — (всегда) | — | `2023` |
| EMA | `StaticEmaEnabled` / `DynamicEmaEnabled` | `*EmaAlpha` | `2024` |

Аудит пишется на каждое срабатывание, без троттлинга. Очередь аудита кольцевая: в аварийном
режиме она забивается — это принятое поведение, а не дефект. Аудит `2024` (EMA) пишется один
раз при создании пайплайна, а не на кадр.

### Решения

**Дрейф не фильтруется.** Медленный синус периодом ~2 с неотличим от реальной нагрузки:
вагон, наезжающий на платформу, даёт такое же плавное изменение кода. Любой порог, который
поймал бы дрейф, резал бы легитимное взвешивание. Отвергнуто в пользу клэмпа и дельты,
которые ловят быстрые аномалии, а не медленные.

**Детектора застревания нет в статике.** При взвешивании с расцепкой под башмак вагон законно
стоит неподвижно, и постоянный код — признак нормального процесса, а не неисправности.
В динамике поток идёт непрерывно и вагон движется, поэтому там та же серия одинаковых кодов
однозначно означает неисправность.

Состояние связи считается активным только пока приходят валидные кадры/сэмплы.
Таймаут, после которого драйвер переводит состояние в `ConnectionChanged(false)`, задаётся
вызывающей формой через `ConnectionTimeoutMs`, а не зашит в самом драйвере:

| Где | Таймаут |
| --- | --- |
| `StaticWeighingForm` | `2000` мс |
| `DynamicWeighingForm` | `5000` мс |
| `ServiceForm`, вкладки статики (`Сервисный режим Статика`, `Калибровка Статика`) | `1000` мс |
| `ServiceForm`, вкладки динамики (`Сервисный режим Динамика`, `Калибровка Динамика`) | `5000` мс |

## Форсированное закрытие порта

`SerialPort.Close()` у .NET может зависать без таймаута (минуты), если устройство на другом
конце линии не отвечает — внутри `SerialStream.Dispose()` идёт безусловное ожидание фонового
overlapped-цикла `WaitCommEvent`. `Services/Hardware/SerialPortForceCloser.cs` оборачивает
`Close()` ограниченным ожиданием (500 мс) и, если не уложился, форсированно обрывает pending
I/O через `CancelIoEx` на сыром хендле порта. Оба драйвера (`SimA04ReaderStatic`,
`SimA04ReaderDynamic`) используют этот хелпер в своём `Close()` и выставляют `IsPoisoned = true`,
если пришлось форсировать — такой экземпляр дальше не переиспользуется для повторного `Open()`.

## Параметры порта

- baud rate: `4800`;
- parity: `Even`;
- data bits: `8`;
- stop bits: `One`;
- poll byte по умолчанию для статического режима: `214`;
- статический ответ: 4 байта;
- динамический режим `Set=126`, `Req=254`: непрерывный поток 5-байтовых сэмплов, подробности в `protocol.md`.

## Формат статического кадра

Кадр АЦП представлен моделью `SimA04Frame`.

Ожидаемая длина - ровно 4 байта. Если длина отличается, кадр считается
невалидным.

Расчет каналов:

```text
CH0 = byte[1] * 256 + byte[0]
CH1 = byte[3] * 256 + byte[2]
```

Каналы используются так:

- `ActiveChannel.Main` - основной канал, `CH0`;
- `ActiveChannel.Backup` - резервный канал, `CH1`.

## События `SimA04ReaderStatic`

- `FrameReceived` - валидный распарсенный кадр.
- `RawFrameReceived` - сырые байты, используются для диагностики в сервисной форме и аудите.
- `ConnectionChanged` - изменение состояния COM-соединения.
- `IsPortOpen` - COM-порт открыт программой.
- `IsConnected` - за последнее время приходили валидные кадры от АЦП.

## События `SimA04ReaderDynamic`

- `SampleReceived` - валидный распарсенный 5-байтовый сэмпл.
- `RawSampleReceived` - сырые 5 байт сэмпла.
- `ConnectionChanged` - изменение состояния COM-соединения.
- `IsPortOpen` - COM-порт открыт программой.
- `IsConnected` - за последнее время приходили валидные сэмплы от АЦП.

## Статическая калибровка

Статический расчет выполняется по активным точкам текущего канала.

Алгоритм:

1. выбрать активные точки канала;
2. отсортировать по `adc_code`;
3. выбрать последнюю точку, код которой меньше или равен текущему коду;
4. если код ниже первой точки, использовать первую;
5. если код выше последней, использовать последнюю;
6. посчитать вес как:

```text
тонны = текущий_код * (масса_точки / код_точки)
```

Интерполяция между точками не используется.

## Динамическая калибровка

Динамический режим использует коэффициенты из `calibration_dynamic`:

```text
тонны = K * код_АЦП
```

Если коэффициент выбранного направления равен `0`, форма использует тестовый fallback по НПВ из настроек:

```text
тонны = код_АЦП * MaxCapacityTonnes / 65535
```

Это нужно для проверки динамической формы с эмулятором до настройки реальной динамической калибровки.

Выбор коэффициента зависит от направления:

- направление начинается с `→` - `k_plus`;
- иначе - `k_minus`.

## Установка нуля

Оператор может установить ноль только если текущий вес находится в пределах
`OperatorZeroLimitPercent` от НПВ. Предел в тоннах вычисляется как:

```text
MaxCapacityTonnes * OperatorZeroLimitPercent / 100
```

Смещение нуля хранится в памяти формы взвешивания и применяется к текущим
показаниям до округления.

## Эмулятор `ScaleListener`

Проект `ScaleListener` нужен для тестирования обмена по COM-порту.

Особенности:

- открывает `COM4`;
- принимает poll-байт `214`;
- генерирует 4-байтовый ответ для двух каналов;
- переводит заданный вес в код АЦП по диапазону `0..65535`;
- может добавлять случайную погрешность в тоннах;
- показывает лог входящих и исходящих байтов.

Для локального тестирования обычно нужна пара виртуальных COM-портов: один порт
указывать в `settings.json` основного приложения, второй - использовать
эмулятором.
