# Аппаратная интеграция

## АЦП "СИМ А04"

Статический драйвер находится в `Services/Hardware/SimA04ReaderStatic.cs`.
Динамический драйвер находится в `Services/Hardware/SimA04ReaderDynamic.cs`.

Статический драйвер работает по циклу:

1. открыть COM-порт;
2. отправить один poll-байт;
3. принять ответ;
4. подождать 20 мс после последнего принятого байта;
5. разобрать накопленный буфер как кадр;
6. опубликовать события;
7. отправить следующий poll-байт.

## Обработка входного потока (план — реализован только парсинг)

Эта часть описывает целевой pipeline после чтения байтов из COM-порта. Это план на
следующий шаг, не текущее поведение: сейчас реализован только шаг 2 (парсинг) через
`SimA04Frame.Parse` (`Models/SimA04Frame.cs`) — проверяется только длина пакета (ровно
4 байта), без синхронизирующего окна, клэмпа, delta-фильтра и EMA-сглаживания. Коды АЦП
пока идут в `CalibrationCalculator` как есть.

1. Синхронизация потока
   - скользящее окно 4 байта;
   - ресинк после битого пакета.

2. Парсинг двух каналов `UInt16` little-endian
   - `CH0 = byte[1] * 256 + byte[0]`;
   - `CH1 = byte[3] * 256 + byte[2]`;
   - разбор только после набора полного кадра.

3. Hard clamp `[min, max]`
   - значение вне диапазона считается невалидным;
   - выброс за диапазон приводит к дропу кадра.

4. Delta filter
   - если `|Δ от предыдущего|` больше порога, кадр дропается;
   - подавляет всплески и дребезг.

5. EMA сглаживание
   - коэффициент `α` подбирается под частоту пакетов;
   - используется для стабилизации отображаемого значения.

Состояние связи считается активным только пока приходят валидные кадры/сэмплы.
Таймаут, после которого драйвер переводит состояние в `ConnectionChanged(false)`, задаётся
вызывающей формой через `ConnectionTimeoutMs`, а не зашит в самом драйвере:

| Где | Таймаут |
| --- | --- |
| `StaticWeighingForm` | `2000` мс |
| `DynamicWeighingForm` | `5000` мс |
| `ServiceForm`, вкладки статики (`Сервисный режим Статика`, `Калибровка Статика`) | `1000` мс |
| `ServiceForm`, вкладки динамики (`Сервисный режим Динамика`, `Калибровка Динамика`) | `5000` мс |

## Форсированное закрытие порта

`SerialPort.Close()` у .NET может зависать без таймаута (минуты), если устройство на другом
конце линии не отвечает — внутри `SerialStream.Dispose()` идёт безусловное ожидание фонового
overlapped-цикла `WaitCommEvent`. `Services/Hardware/SerialPortForceCloser.cs` оборачивает
`Close()` ограниченным ожиданием (500 мс) и, если не уложился, форсированно обрывает pending
I/O через `CancelIoEx` на сыром хендле порта. Оба драйвера (`SimA04ReaderStatic`,
`SimA04ReaderDynamic`) используют этот хелпер в своём `Close()` и выставляют `IsPoisoned = true`,
если пришлось форсировать — такой экземпляр дальше не переиспользуется для повторного `Open()`.

## Параметры порта

- baud rate: `4800`;
- parity: `Even`;
- data bits: `8`;
- stop bits: `One`;
- poll byte по умолчанию для статического режима: `214`;
- статический ответ: 4 байта;
- динамический режим `Set=126`, `Req=254`: непрерывный поток 5-байтовых сэмплов, подробности в `protocol.md`.

## Формат статического кадра

Кадр АЦП представлен моделью `SimA04Frame`.

Ожидаемая длина - ровно 4 байта. Если длина отличается, кадр считается
невалидным.

Расчет каналов:

```text
CH0 = byte[1] * 256 + byte[0]
CH1 = byte[3] * 256 + byte[2]
```

Каналы используются так:

- `ActiveChannel.Main` - основной канал, `CH0`;
- `ActiveChannel.Backup` - резервный канал, `CH1`.

## События `SimA04ReaderStatic`

- `FrameReceived` - валидный распарсенный кадр.
- `RawFrameReceived` - сырые байты, используются для диагностики в сервисной форме и аудите.
- `ConnectionChanged` - изменение состояния COM-соединения.
- `IsPortOpen` - COM-порт открыт программой.
- `IsConnected` - за последнее время приходили валидные кадры от АЦП.

## События `SimA04ReaderDynamic`

- `SampleReceived` - валидный распарсенный 5-байтовый сэмпл.
- `RawSampleReceived` - сырые 5 байт сэмпла.
- `ConnectionChanged` - изменение состояния COM-соединения.
- `IsPortOpen` - COM-порт открыт программой.
- `IsConnected` - за последнее время приходили валидные сэмплы от АЦП.

## Статическая калибровка

Статический расчет выполняется по активным точкам текущего канала.

Алгоритм:

1. выбрать активные точки канала;
2. отсортировать по `adc_code`;
3. выбрать последнюю точку, код которой меньше или равен текущему коду;
4. если код ниже первой точки, использовать первую;
5. если код выше последней, использовать последнюю;
6. посчитать вес как:

```text
тонны = текущий_код * (масса_точки / код_точки)
```

Интерполяция между точками не используется.

## Динамическая калибровка

Динамический режим использует коэффициенты из `calibration_dynamic`:

```text
тонны = K * код_АЦП
```

Если коэффициент выбранного направления равен `0`, форма использует тестовый fallback по НПВ из настроек:

```text
тонны = код_АЦП * MaxCapacityTonnes / 65535
```

Это нужно для проверки динамической формы с эмулятором до настройки реальной динамической калибровки.

Выбор коэффициента зависит от направления:

- направление начинается с `→` - `k_plus`;
- иначе - `k_minus`.

## Установка нуля

Оператор может установить ноль только если текущий вес находится в пределах
`OperatorZeroLimitPercent` от НПВ. Предел в тоннах вычисляется как:

```text
MaxCapacityTonnes * OperatorZeroLimitPercent / 100
```

Смещение нуля хранится в памяти формы взвешивания и применяется к текущим
показаниям до округления.

## Эмулятор `ScaleListener`

Проект `ScaleListener` нужен для тестирования обмена по COM-порту.

Особенности:

- открывает `COM4`;
- принимает poll-байт `214`;
- генерирует 4-байтовый ответ для двух каналов;
- переводит заданный вес в код АЦП по диапазону `0..65535`;
- может добавлять случайную погрешность в тоннах;
- показывает лог входящих и исходящих байтов.

Для локального тестирования обычно нужна пара виртуальных COM-портов: один порт
указывать в `settings.json` основного приложения, второй - использовать
эмулятором.
