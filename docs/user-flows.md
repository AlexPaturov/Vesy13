# Пользовательские сценарии

## Главное меню

`MainForm` открывает основные режимы:

- статическое взвешивание;
- динамическое взвешивание;
- сервис;
- корректировки и перенос;
- печать;
- журналы.

Пока открыта дочерняя форма, главное окно скрывается. После закрытия дочерней
формы главное окно снова отображается.

## Статическое взвешивание

Форма: `StaticWeighingForm`.

Сценарий:

1. При открытии форма подписывается на события АЦП.
2. Оператор нажимает `ВЗВЕСИТЬ` или пробел для фиксации первой тележки.
3. Форма сохраняет код АЦП первой тележки и переходит в состояние ожидания второй.
4. Оператор снова нажимает `ВЗВЕСИТЬ` или пробел.
5. Форма рассчитывает вес обеих тележек, общий вес и добавляет строку в таблицу.
6. Запись сохраняется в PostgreSQL через `LocalRepository.SaveWagonAsync`.
7. После завершения состава оператор нажимает `Завершить`.

Если форма закрывается при незавершенном вагоне, оператор получает
предупреждение.

## Динамическое взвешивание

Форма: `DynamicWeighingForm`.

Сценарий:

1. Оператор выбирает направление движения состава.
2. Оператор фиксирует первую тележку.
3. Оператор фиксирует вторую тележку.
4. Вес рассчитывается через динамический коэффициент выбранного направления.
5. Запись сохраняется в PostgreSQL с режимом `ДИНАМИКА` и направлением.

Без выбранного направления форма не разрешает взвешивание и установку нуля.

## Перенос локальной записи

Форма: `CorrectionsForm`.

Сценарий:

1. Форма загружает неперенесенные записи из PostgreSQL.
2. Оператор выбирает локальную запись.
3. Приложение проверяет превышение НПВ.
4. Оператор вводит номер вагона и реквизиты.
5. Оператор выбирает `GPRI` или `GRAS`, брутто или тару.
6. При переносе создается запись Firebird.
7. После успешной вставки локальная запись помечается как перенесенная.

Если вставка в Firebird прошла успешно, но PostgreSQL не смог пометить запись
как перенесенную, пользователь получает отдельное предупреждение.

## Корректировка Firebird-записи

Форма: `CorrectionsForm`.

Сценарий:

1. Оператор выбирает запись из списка Firebird.
2. Форма заполняет верхнюю панель текущими реквизитами.
3. Оператор изменяет номер вагона, груз, тару, потребителя, плательщика или цех.
4. При сохранении вызывается `FactoryRepository.UpdateAsync`.
5. Событие записывается в аудит.

Для выбора тары форма может подгрузить варианты из `GPRI` и `GRAS` за последний
год по номеру вагона.

## Печать отвесной

Форма: `PrintForm`.

Сценарий:

1. Оператор выбирает таблицу `GPRI` или `GRAS`.
2. Указывает период и дополнительные фильтры: вагон, груз, потребитель, документ.
3. Нажимает поиск.
4. Выбирает строки для печати или оставляет выбор пустым, чтобы печатать все найденные.
5. Приложение создает PDF через `WeighingSlipDocument`.
6. PDF открывается средствами ОС.

PDF сохраняется во временную папку с именем вида `otves_yyyyMMdd_HHmmss.pdf`.

## Сервис и калибровка

Форма: `ServiceForm`.

Основные действия:

- подключение и мониторинг COM-порта;
- просмотр сырых кадров;
- выбор канала `CH0`/`CH1`;
- добавление, удаление и сохранение статических калибровочных точек;
- расчет линейной формулы методом наименьших квадратов;
- захват коэффициентов динамического режима;
- сохранение настроек приложения;
- вход администратора и смена пароля.

## Журнал

Форма: `LogsForm`.

Сценарий:

1. Оператор выбирает период.
2. Форма загружает записи из `audit_log`.
3. При необходимости журнал экспортируется в CSV.
